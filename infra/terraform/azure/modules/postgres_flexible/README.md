# Module: postgres_flexible

**Status:** implemented in Block 4.16 — applied in Block 4.17 — Alembic migrated in Block 4.18
— gated by `create_postgres` (default `false`, set to `true` in `terraform.postgres.example.tfvars`)
in `environments/dev`.

## Purpose

Provision Azure Database for PostgreSQL Flexible Server plus the `fittrack_ai`
application database, replacing the local Docker Compose Postgres for the deployed API.
The connection string this module produces becomes the real `DATABASE-URL` secret in
Key Vault (replacing the placeholder used since Block 4.15).

## Architecture

```text
Terraform random_password → PostgreSQL Flexible Server → fittrack_ai database
                                                      ↓
                                            database_url (sensitive output)
                                                      ↓
                                            Key Vault DATABASE-URL secret
```

| Name | Sensitive | Description |
|---|---|---|
| `server_id` | no | PostgreSQL Flexible Server resource ID. |
| `server_name` | no | Server name. |
| `fqdn` | no | Server FQDN for connection strings. |
| `database_name` | no | Application database name. |

## Resources created

- `random_password.administrator` — 24-char password with safe special characters
- `azurerm_postgresql_flexible_server.this` — Flexible Server (public access, no VNet)
- `azurerm_postgresql_flexible_server_database.this` — `fittrack_ai` database
- `azurerm_postgresql_flexible_server_firewall_rule.this` — optional, empty by default

Costs vary by region and usage. Destroy the server when not needed to avoid charges.

## Security decisions

1. **Password generated by Terraform** — never committed; marked `sensitive` in outputs.
2. **No broad firewall rules** — `allowed_firewall_rules` defaults to `{}`; no `0.0.0.0/0`.
3. **Public access is temporary** — acceptable for dev/demo; private VNet access planned for hardening.
4. **SSL required** — connection string includes `sslmode=require`.
5. **No secret values in environment outputs** — only server name, FQDN, and database name.

## Out of scope

VNet, subnets, Private DNS, Private Endpoint, HA, replicas, connection pooling,
additional users, extensions, migration jobs, Alembic execution.

### Block 4.18 — Alembic migration (completed)

- Firewall rule `temp-local-alembic` added via Azure CLI (single local IP), not Terraform.
- `DATABASE_URL` retrieved from Key Vault; password and connection string never exposed.
- Rule removed after migration; `postgres_allowed_firewall_rules = {}` unchanged in Terraform.
- Private networking deferred to a future hardening block.

To repeat migrations safely:

```bash
# 1. Add temporary firewall rule for your public IP (Azure CLI example)
az postgres flexible-server firewall-rule create \
  --resource-group rg-fittrack-ai-dev \
  --server-name psql-fittrack-ai-pg-dev01 \
  --name temp-local-alembic \
  --start-ip-address "<YOUR_PUBLIC_IP>" \
  --end-ip-address "<YOUR_PUBLIC_IP>"

# 2. Load DATABASE_URL from Key Vault (do not echo the value)
export DATABASE_URL="$(az keyvault secret show \
  --vault-name kvfittrackaidevdev01 --name DATABASE-URL --query value --output tsv)"
test -n "$DATABASE_URL" && echo "DATABASE_URL loaded"

# 3. Run migrations
cd backend && uv run alembic upgrade head

# 4. Remove temporary firewall rule
az postgres flexible-server firewall-rule delete \
  --resource-group rg-fittrack-ai-dev \
  --server-name psql-fittrack-ai-pg-dev01 \
  --name temp-local-alembic --yes
```
